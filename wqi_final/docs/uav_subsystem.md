# Standalone Campus UAV Subsystem

## Scope

This stage runs the UAV independently from the UGV. The UAV starts at the
logistics-center pad, takes off, hovers, follows a low-altitude campus air
corridor to one or more delivery pads, simulates delivery, returns home, and
lands. The UAV launch does not spawn the UGV or start UGV localization,
planning, or control nodes. A lightweight map server is started only for RViz.

## Architecture

| Package | Responsibility |
|---|---|
| `uav_interfaces` | Flight/delivery actions and mission energy-check service |
| `uav_description` | Quadrotor, sensors, collisions, TF, and RViz config |
| `uav_control` | Flight, safety, battery consumption, dock charging and state monitoring |
| `uav_navigation` | Delivery pads, 15 m corridor graph, airspace rules, RViz markers |
| `uav_application` | Delivery mission action server and state machine |
| `uav_bringup` | Standalone Gazebo, map server, UAV nodes, and optional RViz |
| `sjtu_drone_description` | Vendored Gazebo force/torque dynamics plugin |

All public interfaces use the `/uav` namespace. Source files are under
`simulation_ws/src`; `build`, `install`, and `log` are generated by ROS tools.

## Dynamics And Collision

The vehicle uses the ROS2 branch of
[`NovoG93/sjtu_drone`](https://github.com/NovoG93/sjtu_drone/tree/ros2). The
Gazebo plugin drives the rigid body with force and torque. Flight is not
implemented by repeatedly setting the entity pose.

The physical collision model covers the body, crossed arms, four rotor disks,
lidar mast, and two landing skids. Controlled landing holds XY and continues
until the skids reach the ground. The larger safety sphere is a non-contact
perception envelope, not a physical Gazebo sphere, so it does not interfere
with landing or future UGV docking.

## Sensor Stack

The UAV uses the following sensors:

| Sensor | Public topic | Type |
|---|---|---|
| Top 3D lidar | `/uav/lidar/points` | `sensor_msgs/msg/PointCloud2` |
| Down-view RGB camera | `/uav/down_camera/image_raw` | `sensor_msgs/msg/Image` |
| Downward range | `/uav/range/down` | `sensor_msgs/msg/Range` |
| Front-down range | `/uav/range/front_down` | `sensor_msgs/msg/Range` |
| Rear-down range | `/uav/range/rear_down` | `sensor_msgs/msg/Range` |
| Left-down range | `/uav/range/left_down` | `sensor_msgs/msg/Range` |
| Right-down range | `/uav/range/right_down` | `sensor_msgs/msg/Range` |
| IMU | `/uav/imu` | `sensor_msgs/msg/Imu` |

The top lidar runs at 10 Hz with 240 horizontal by 24 vertical rays. Its
horizontal field is 360 degrees and its elevation field is `-45 deg` to
`+89 deg`. This keeps the total at 5760 rays while covering diagonal-down,
horizontal, upper, and near-vertical-up directions. Empty sky produces no
point-cloud return; it is not converted into artificial obstacle points.

Gazebo Classic can publish an out-of-range value when a `Range` ray has no
hit. The model therefore publishes five internal `LaserScan` streams under
`/uav/range_raw/*`. `safety_monitor` reduces each cone to its nearest valid
ray, clamps no-return to the configured maximum range, and publishes the
standard public `Range` topics listed above.

The down camera is RGB only. A depth camera is intentionally not included to
keep VirtualBox load low.

## Frames

```text
map
└── uav/odom
    └── uav/base_footprint
        └── uav/base_link
            ├── uav/imu_link
            ├── uav/top_3d_lidar_link
            ├── uav/down_camera_link
            │   └── uav/down_camera_optical_frame
            ├── uav/down_range_link
            ├── uav/front_down_range_link
            ├── uav/rear_down_range_link
            ├── uav/left_down_range_link
            └── uav/right_down_range_link
```

The Gazebo dynamics plugin is the only publisher of the dynamic
`uav/odom -> uav/base_footprint` transform. It publishes normalized frame IDs
without leading slashes, and the RViz RobotModel applies the UAV namespace
prefix to the unprefixed URDF link names. In the RViz property this is written
as `uav` without a trailing slash because RViz inserts the separator. All UAV
nodes use simulation time.

## Safety Control

`safety_monitor` fuses the 3D point cloud and four diagonal range sensors into
a body-centered 1.8 m safety sphere. It applies:

- a 0.58 m UAV self-return filter;
- flat-ground filtering using the downward range sensor;
- diagonal ground-return filtering during takeoff and landing;
- stale-sensor detection;
- a green or red `/uav/safety_sphere` RViz marker.

The public safety topics are:

| Topic | Purpose |
|---|---|
| `/uav/safety/blocked` | Combined collision-envelope state |
| `/uav/safety/status` | Clear, blocked sensor, or stale-sensor reason |
| `/uav/safety/min_distance` | Nearest filtered obstacle distance |
| `/uav/safety/ground_clearance` | Downward measured clearance |
| `/uav/safety_sphere` | RViz sphere marker |

When the sphere is blocked, `FlyToPose` locks the first detected hold position.
It resumes if the obstacle clears and aborts with a reason if it remains for
10 seconds. A flight segment succeeds only after both position error and
linear speed (`<= 0.1 m/s`) remain settled for one second. This prevents
residual velocity from carrying the UAV outside a corridor after a right-angle
turn.

## Flight Strategy

The cruise altitude is 15 m. Campus buildings can be taller than the UAV, so
horizontal flight never uses a direct line between arbitrary delivery pads.
`uav_delivery_waypoints.yaml` defines a connected, closed corridor graph that
follows the campus road network and includes small building-corner offsets.
The library UAV pad is on the east corner of the north-campus-loop roadway. It
does not use the road center inside the narrow five-meter gap between the
laboratory and library. All 40 nodes and 41 edges remain outside every building
footprint expanded by the 1.8 m safety radius. The library return route first
moves east to `x=78`, reaches `y=65`, and turns south only after reaching
`x=42`. These offsets leave control margin for physical turning drift beside
the east, north, and west library walls.

RViz separates route intent from vehicle dynamics. `/uav/planned_path` is a
cyan, transient-local path made of straight corridor segments, while
`/uav/path` is the orange odometry trace and can show small physical turn arcs.
The UAV model follows the same `map -> uav/odom -> uav/base_link` chain as both
paths.

The mission sequence is:

1. Take off vertically and stabilize at 3 m for five seconds.
2. Climb vertically to 15 m.
3. Follow shortest-path corridor segments to the target.
4. Approach vertically to the target's configured floor altitude.
5. Hover for five seconds and publish the delivery event.
6. Climb to 15 m, follow the corridor home, approach, and land.

Mission phases are `IDLE`, `TAKEOFF`, `HOVER`, `CRUISE`, `APPROACH`,
`DELIVERING`, `RETURNING`, `LANDING`, `COMPLETED`, and `FAILED`.

## Build And Run

```bash
cd ~/design_final/wqi_final/simulation_ws
source /opt/ros/humble/setup.bash
colcon build --packages-select \
  sjtu_drone_description uav_interfaces uav_description uav_control \
  uav_navigation uav_application uav_bringup
source install/setup.bash
ros2 launch uav_bringup uav_sim.launch.py \
  gui:=false rviz:=true visualize_sensor_rays:=false
```

Send a mission from another sourced terminal:

```bash
ros2 action send_goal /uav/execute_delivery \
  uav_interfaces/action/ExecuteDelivery \
  "{targets: ['teaching_building'], return_home: true}" --feedback
```

Use `gui:=true rviz:=false visualize_sensor_rays:=true` to inspect Gazebo lidar
rays. The switch controls only the blue ray graphic; sensor topics remain
active when it is `false`. Keep only one 3D viewer active in VirtualBox.

## Validation

The current source was validated in the campus world with headless Gazebo:

- Xacro and URDF parsing passed.
- Eighteen RViz-prefix, waypoint, mission-state, and safety-geometry tests
  passed.
- UAV Python source passed `ament_flake8`.
- The five public range topics remained valid at ground level and at 15 m;
  high-altitude no-return values no longer crash rclpy.
- Ground-level range samples were approximately `0.088 m` down and
  `0.141-0.156 m` for the four diagonal sensors.
- The configured lidar elevation is `-45 deg` to `+89 deg`; at the logistics
  pad, actual campus obstacle returns reached `+77.3 deg`. The remaining upper
  cone contained empty sky.
- The 1.8 m expanded-building corridor check reported zero conflicts.
- Waypoint tests verify that each target has a valid delivery floor and altitude.
  The current library door target is `(62, 40.5, 8.0)` for floor 3; the teaching
  target is `(-40, 9.5, 8.0)` for floor 3.

## Current Limitations

- Localization uses Gazebo ground truth; UAV SLAM and GPS/IMU fusion are not
  included.
- Known static buildings are avoided by the corridor graph. An unexpected
  obstacle causes hover/resume or abort, not online 3D detour planning.
- Delivery is a ROS event rather than a physical package gripper.
- Battery power uses the Zeng/Gong/Dai propulsion equations, dynamic package
  mass, and acceleration-aware route integration rather than ESC or
  electrochemical telemetry. Safe-return admission and UGV dock charging are
  implemented;
  unexpected in-flight battery hardware faults are not.
- The standalone launch does not start the UGV. UGV transport, fixed-point
  rendezvous, platform landing, and task handoff are implemented by the
  separate `cooperative_delivery` package.

The cooperative layer preserves `/uav/execute_delivery` as the UAV boundary
and does not couple the UGV and UAV low-level controllers directly. See
[`cooperative_delivery.md`](cooperative_delivery.md).
Battery equations, topics and verification commands are documented in
[`uav_battery.md`](uav_battery.md).
